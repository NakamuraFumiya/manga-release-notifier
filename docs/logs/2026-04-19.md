## 学んだこと

- **継承より合成 (Composition over Inheritance)**: Gang of Four の原則。Go は継承を持たないので言語レベルで強制されている
- **Go のコンポジション**: 構造体の埋め込み（has-a）とインターフェースによる多態性。ストラテジーパターンは Ruby でも Go でも同じ構造で書ける
- **OOP と言語の関係**: Ruby は Smalltalk 寄りの純粋 OOP、Go は継承なしだが「継承より合成」を忠実に実装。EC 等のマイクロサービス境界では Go も OOP 的な設計に十分向く
- **Lambda 関連のパッケージ分け**
  - `aws-lambda-go`: handler を Lambda ランタイムに登録する（Lambda 内で動くコード）
  - `aws-sdk-go-v2`: AWS サービスを叩くためのクライアント（呼び出し側）
  - Phase 1 では前者だけあれば十分
- **handler シグネチャの選択肢**: `func()`, `func() error`, `func(ctx) error`, `func(ctx, event) (result, error)`
- **`lambda.Start()` はランタイム待ちループ**: ローカルで `go run` しても動かない。単体テストで handler 関数を直接呼ぶのが確実
- **LocalStack のポート構造**: 4566 は Gateway（全サービスの受付口）、4510-4559 は内部サービス用。ホストから invoke するだけなら 4510-4559 のホスト公開は不要だが、公式推奨なので残す
- **go.mod の `// indirect`**: 直接 import しているのに付いていたら `go mod tidy` で整う

### 手続き型 vs OOP の対比

ADR-013 で使った「手続き型寄りに書く」という表現を復習用にまとめる。

**手続き型 (procedural)**

- データと手続き（関数）が **分離** している
- 関数が主役、データは引数として渡される二次的なもの
- 処理の流れをそのまま書く（C 言語が典型）

**OOP**

- データと振る舞いを一つの単位（オブジェクト）に **まとめる**（カプセル化）
- オブジェクト自身が操作を持つ
- インターフェースで多態性を表現

**Go での具体例: 漫画の最新話を取得して通知する**

手続き型寄り:

```go
type Manga struct {
    Title    string
    Episodes []Episode
}

// 全部 package-level 関数として定義
func GetLatestEpisode(m Manga) Episode { ... }
func IsNewEpisode(m Manga, latest Episode) bool { ... }
func NotifyToDiscord(webhookURL string, m Manga, ep Episode) error { ... }

func main() {
    m := loadManga()
    latest := GetLatestEpisode(m)
    if IsNewEpisode(m, latest) {
        NotifyToDiscord("https://...", m, latest)
    }
}
```

- `Manga` はただのデータ入れ物
- 通知先を変えたい時は `NotifyToSlack`, `NotifyToEmail` を並べて if/switch で振り分け

OOP 寄り:

```go
type Manga struct {
    title    string
    episodes []Episode
}

// メソッドレシーバーで Manga 自身に振る舞いを持たせる
func (m *Manga) LatestEpisode() Episode { ... }
func (m *Manga) HasNewEpisode() bool { ... }

// 通知先を抽象化
type Notifier interface {
    Notify(m *Manga, ep Episode) error
}

type DiscordNotifier struct{ webhookURL string }
func (d DiscordNotifier) Notify(m *Manga, ep Episode) error { ... }

type SlackNotifier struct{ token string }
func (s SlackNotifier) Notify(m *Manga, ep Episode) error { ... }

func main() {
    m := loadManga()
    if m.HasNewEpisode() {
        var notifier Notifier = DiscordNotifier{webhookURL: "..."}
        notifier.Notify(m, m.LatestEpisode())
    }
}
```

- `Manga` 自身が「最新話を返す」「新着か判定する」責務を持つ
- `Notifier` インターフェースで通知先を抽象化 → 追加時は新しい型を作るだけ
- フィールドは小文字（カプセル化）、公開メソッド経由でアクセス

**本質的な違い**

| 観点 | 手続き型 | OOP |
|---|---|---|
| データと処理 | 分離 | 一体化（カプセル化） |
| 振る舞いの切り替え | if/switch | インターフェース（多態性） |
| 変更時の影響範囲 | 関数を書き換える | 新しい型を追加するだけ |
| テスト | 関数単体 | モック差し替え |

**Go は中間言語**

Go は手続き型でも OOP でも書ける。初心者は package-level 関数で済ませて手続き型寄りになりがち。Phase 1 の `hello(ctx context.Context) error` も現状は手続き型寄り（ドメインがないので当然だが）。

本プロジェクトで「OOP を意識する」とは、**Phase 3-4 でドメインが出てきた時に、メソッドとインターフェースで振る舞いを表現する**ということを意味する。

## 深掘りしたいこと

- **OOP の 4 本柱と SOLID の各原則**: 今日は概要のみ。`docs/oop-principles.md` のプレースホルダを Phase 進行に合わせて埋める。参考: Robert C. Martin『Clean Architecture』
- **Go のインターフェース設計思想**: 暗黙実装、「小さいインターフェース」文化、`io.Reader` / `io.Writer` の設計意図。参考: [Effective Go - Interfaces](https://go.dev/doc/effective_go#interfaces), 標準ライブラリのソース
- **Lambda handler シグネチャの選び方**: context のタイムアウト制御、EventT 型定義、panic の扱い。参考: [AWS公式 Lambda ハンドラー (Go)](https://docs.aws.amazon.com/ja_jp/lambda/latest/dg/golang-handler.html), `aws-lambda-go` の README
- **LocalStack の Lambda 実行モデル**: Lambda 実行時に spawn されるコンテナの仕組み、Docker network 経由の通信、`LAMBDA_DOCKER_NETWORK` 設定の意味。参考: LocalStack 公式ドキュメント
- **Go workspaces (`go.work`)**: 今回は単一 go.mod で十分だが、複数モジュール構成がいつ有効になるか
