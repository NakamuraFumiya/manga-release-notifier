## 学んだこと

- **継承より合成 (Composition over Inheritance)**: Gang of Four の原則。Go は継承を持たないので言語レベルで強制されている
- **Go のコンポジション**: 構造体の埋め込み（has-a）とインターフェースによる多態性。ストラテジーパターンは Ruby でも Go でも同じ構造で書ける
- **OOP と言語の関係**: Ruby は Smalltalk 寄りの純粋 OOP、Go は継承なしだが「継承より合成」を忠実に実装。EC 等のマイクロサービス境界では Go も OOP 的な設計に十分向く
- **Lambda 関連のパッケージ分け**
  - `aws-lambda-go`: handler を Lambda ランタイムに登録する（Lambda 内で動くコード）
  - `aws-sdk-go-v2`: AWS サービスを叩くためのクライアント（呼び出し側）
  - Phase 1 では前者だけあれば十分
- **handler シグネチャの選択肢**: `func()`, `func() error`, `func(ctx) error`, `func(ctx, event) (result, error)`
- **`lambda.Start()` はランタイム待ちループ**: ローカルで `go run` しても動かない。単体テストで handler 関数を直接呼ぶのが確実
- **LocalStack のポート構造**: 4566 は Gateway（全サービスの受付口）、4510-4559 は内部サービス用。ホストから invoke するだけなら 4510-4559 のホスト公開は不要だが、公式推奨なので残す
- **go.mod の `// indirect`**: 直接 import しているのに付いていたら `go mod tidy` で整う

## 詰まったところ・疑問

- `aws-sdk-go-v2/service/lambda` を追加しようとしたが、これは「Lambda を外から呼ぶ側」の SDK。handler 実装には不要だった
- AWS 公式の Go handler ドキュメントの例は S3 操作を含む複合例で、Phase 1 の最小構成には過剰だった
- Docker Desktop UI では LocalStack のポート公開が部分的にしか見えなかったが、`docker ps` で確認すると 4566 もちゃんと公開されていた（UI の表示簡略化が原因）

## 深掘りしたいこと

- **OOP の 4 本柱と SOLID の各原則**: 今日は概要のみ。`docs/oop-principles.md` のプレースホルダを Phase 進行に合わせて埋める。参考: Robert C. Martin『Clean Architecture』
- **Go のインターフェース設計思想**: 暗黙実装、「小さいインターフェース」文化、`io.Reader` / `io.Writer` の設計意図。参考: [Effective Go - Interfaces](https://go.dev/doc/effective_go#interfaces), 標準ライブラリのソース
- **Lambda handler シグネチャの選び方**: context のタイムアウト制御、EventT 型定義、panic の扱い。参考: [AWS公式 Lambda ハンドラー (Go)](https://docs.aws.amazon.com/ja_jp/lambda/latest/dg/golang-handler.html), `aws-lambda-go` の README
- **LocalStack の Lambda 実行モデル**: Lambda 実行時に spawn されるコンテナの仕組み、Docker network 経由の通信、`LAMBDA_DOCKER_NETWORK` 設定の意味。参考: LocalStack 公式ドキュメント
- **Go workspaces (`go.work`)**: 今回は単一 go.mod で十分だが、複数モジュール構成がいつ有効になるか
